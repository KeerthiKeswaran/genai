```sql
WITH CustomerSpending AS (
    SELECT
        oi.customer_id,
        SUM(oi.quantity * oi.unit_price) AS total_spent,
        COUNT(DISTINCT i.product_id) AS unique_products_bought
    FROM orders o
    INNER JOIN order_items oi ON o.order_id = oi.order_id
    WHERE o.order_date >= CURRENT_DATE - INTERVAL '30 days'
    GROUP BY oi.customer_id
), RankedCustomers AS (
    SELECT 
        cs.customer_id,
        total_spent,
        unique_products_bought,
        RANK() OVER(ORDER BY total_spent DESC) as rank
    FROM CustomerSpending cs
)
SELECT *
FROM RankedCustomers
WHERE rank <= 5;
```
In this query:

1. We first create a Common Table Expression (CTE) called `CustomerSpending` to calculate the total spending and count unique products bought by each customer in the last 30 days using an inner join between orders and order_items tables, where we consider only those orders placed within the last 30 days.
2. We then create another CTE named `RankedCustomers`, which calculates a rank for each row based on total spending (in descending order) in the same query using the RANK() function provided by SQL window functions, ordered by the sum of (`quantity * unit_price`).
3. In our final SELECT statement, we filter out only those customers with `rank` less or equal to 5 and get their details along with total spending & unique products bought columns from CTEs above.
4. We assume here that all column names are correctly indexed for optimal performance; if not, appropriate indexing strategies would be:
    - Create a non-clustered index on `order_date` in the orders table to efficiently filter out rows with relevant dates using an efficient WHERE clause (e.g., `ORDERS(order_date)`). The use of CTEs should also help avoid full table scans for this range query, as filtering is done within each subquery stage which can utilize these indexes.
    - Create a non-clustered index on `(customer_id, total_amount)` in the orders table if we needed to optimize ORDERS(total_amount) or another column that might be frequently queried with `customer_id`. Although not used directly here since only spending is of interest for this query.
    - Create a non-clustered index on `(order_id, product_id)` in the order_items table to quickly join and group by unique products per customer efficiently if needed (but again we are focusing purely on sums which can use all rows from `ORDERS(total_amount)`. If individual item counts matter for each row of result data or further analysis outside this query, a separate count aggregation at the end might be necessary.
    - Lastly, clustering index by customer ID may speed up joins if customers are frequently joined on in queries like these (e.g., `orders(customer_id)`, but given we're using GROUP BY and no need for individual item lookups here per user/order pair).

Be aware that complex window functions, such as RANK(), can be resource-intensive; especially on large datasets - if they become a performance issue or exceed the maximum allowed partitions in SQL Server (1000) by default without proper partitioning of data across tables and/increased buffer sizes might also help.

7. Note that SQL Server supports `RANK()` but there are also alternative ranking functions available; depending on the specific DBMS, options like DENSE_RANK(), PERCENT_RANK() or other approaches could be used to handle ties differently (e.g., in case of two customers spending equally).
